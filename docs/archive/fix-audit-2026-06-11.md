## 🔴 VẤN ĐỀ CRITICAL — Phải sửa NGAY trước khi launch

### 1. 🔒 Formspree Integration Có Thể Đang GửI SAI DỮ LIỆU

**File:** `index.html:1067-1078`

```javascript
// HIỆN TẠI — gửi JSON body
const response = await fetch(RSVP_ENDPOINT, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ name, attendance, guestCount, message, ... })
});
```

**Vấn đề:** Formspree mặc định chỉ xử lý `application/x-www-form-urlencoded` hoặc `multipart/form-data`. Khi gửi `application/json`:
- Formspree **có thể reject** hoặc **không parse được** các field
- Trên free tier, Formspree đôi khi chỉ accept form-encoded data
- Kết quả: Người gửi thấy "thành công" (response 200 từ Formspree redirect) nhưng **data thực tế không được lưu**

**Giải pháp:**
```javascript
// Sửa thành FormData (Formspree đảm bảo parse đúng)
const formData = new FormData();
formData.append('name', name);
formData.append('attendance', attendance);
formData.append('guestCount', guestCount);
formData.append('message', message);
formData.append('timestamp', new Date().toISOString());

const response = await fetch(RSVP_ENDPOINT, {
    method: 'POST',
    body: formData  // Không cần Content-Type header — browser tự set boundary
});
```

**Impact:** 🔴 **CAO NHẤT** — Toàn bộ chức năng RSVP có thể không hoạt động. Khách mời gửi nhưng chủ thiệp không nhận được email. Launch mà không fix = tất cả RSVP mất.

### 2. 🔒 Honeypot `_gotcha` Đang GửI Giá TRỊ — Tự BLOCK MÌNH

**File:** `index.html:1076`

```javascript
body: JSON.stringify({
    ...
    _gotcha: ''  // ← ĐANG GỬI EMPTY STRING
})
```

**Vấn đề:** 
- PRD (dòng 332) ghi: `_gotcha: "" // anti-spam field (bỏ trống = legit, Formspree tự reject nếu có giá trị)`
- Nhưng code đang **gửi `_gotcha: ''`** (empty string) trong body
- Formspree kiểm tra: nếu field `_gotcha` có mặt trong request → reject, bất kể giá trị
- Gửi empty string vẫn khiến Formspree thấy field tồn tại → **tự reject submission của chính mình**

**Giải pháp:**
```javascript
// Không gửi _gotcha trong body — Formspree tự check hidden input trong form
// Hoặc bỏ hoàn toàn field này khỏi JS payload
// Honeypot input trong HTML (dòng 813) đã đủ để Formspree check
```

**Impact:** 🔴 **CAO** — Formspree có thể reject mọi submission vì phát hiện "bot" (do chính code gửi `_gotcha`).

### 8. 🏗️ localStorage Key Generation — Có Thể Gây COLLISION Với Tên Việt

**File:** `index.html:1006-1010`

```javascript
function getStorageKey(name) {
    return 'grad_rsvp_' + name.toLowerCase()
        .normalize('NFD').replace(/[̀-ͯ]/g, '')
        .replace(/[^a-z\s]/g, '').trim().replace(/\s+/g, '_');
}
```

**Vấn đề:**
- "Nguyễn Văn An" → `grad_rsvp_nguyen_van_an`
- "Nguyễn Vân An" (sai chính tả) → `grad_rsvp_nguyen_van_an` ← **CÙNG KEY!**
- "Nguyễn Văn Ánh" → `grad_rsvp_nguyen_van_anh`
- Hai người khác nhau nhưng cùng key → **data bị ghi đè**
- `[^a-z\s]` regex strip tất cả ký tự không phải chữ cái Latin → tên có số/dấu gạch sẽ mất phần đó

**Giải pháp:**
```javascript
function getStorageKey(name) {
    // Dùng toàn bộ tên đã normalize, không strip ký tự
    return 'grad_rsvp_' + name.toLowerCase()
        .normalize('NFD').replace(/[\u0300-\u036f]/g, '')
        .trim().replace(/\s+/g, '_');
}
```

### 9. 🔒 XSS Vulnerability Qua URL Parameter `?guest=`

**File:** `index.html:978-981`

```javascript
const name = decodeURIComponent(guest);
if (input && name.length > 0 && name.length < 100) {
    input.value = name;  // ← DIRECT ASSIGNMENT, NO SANITIZATION
}
```

**Vấn đề:**
- `input.value = name` với `value` property là **an toàn** (HTML-escaped automatically)
- Nhưng nếu sau này thay bằng `innerHTML` hoặc `insertAdjacentHTML` → XSS
- `name.length < 100` check nhưng không check ký tự đặc biệt
- Ví dụ: `?guest=<script>alert(1)</script>` → `input.value` sẽ là string literal (safe), nhưng **defense-in depth** thiếu

**Giải pháp:**
```javascript
// Thêm sanitization
const name = decodeURIComponent(guest);
if (input && name.length > 0 && name.length < 100 && !/<script|javascript:|on\w+=/i.test(name)) {
    input.value = name;
}
```

### 10. ⚡ Không Có Request Timeout — User Có Thể Chờ Vô Hạn

**File:** `index.html:1067`

```javascript
const response = await fetch(RSVP_ENDPOINT, { ... });
// ← Không có timeout. Nếu Formspree chậm hoặc down → user đợi mãi
```

**Giải pháp:**
```javascript
async function fetchWithTimeout(url, options, timeout = 15000) {
    const controller = new AbortController();
    const id = setTimeout(() => controller.abort(), timeout);
    try {
        const response = await fetch(url, { ...options, signal: controller.signal });
        clearTimeout(id);
        return response;
    } catch (err) {
        if (err.name === 'AbortError') {
            throw new Error('Timeout: Vui lòng thử lại');
        }
        throw err;
    }
}
```

### 11. 🏗️ Confetti Memory Leak — 60 DOM Elements Mỗi Lần Submit

**File:** `index.html:1114-1134`

```javascript
function createConfetti() {
    const container = document.createElement('div');
    // ... tạo 60 div elements ...
    setTimeout(() => container.remove(), 3000);  // ← Cleanup sau 3s
}
```

**Vấn đề:**
- 60 divs × 60fps animation = 3600 DOM operations/giây trong 1.5s
- Trên mobile low-end, có thể drop frames
- `setTimeout(() => container.remove(), 3000)` không cancel được nếu user navigate đi
- Nếu user submit 2 lần (thất bại → thử lại), confetti từ lần trước vẫn chạy animation

**Giải pháp:**
```javascript
// Track confetti container để cleanup khi cần
let activeConfetti = null;

function createConfetti() {
    if (activeConfetti) activeConfetti.remove(); // Cleanup previous
    
    const container = document.createElement('div');
    // ... create pieces ...
    activeConfetti = container;
    document.body.appendChild(container);
    
    setTimeout(() => {
        container.remove();
        if (activeConfetti === container) activeConfetti = null;
    }, 3000);
}
```

### 13. 📝 Duplicate `role="main"` — Accessibility Violation

**File:** `index.html:759, 777, 873`

```html
<div id="screen-envelope" class="screen active" role="main" ...>
<div id="screen-invite" class="screen" role="main" ...>      <!-- DUPLICATE! -->
<div id="screen-thanks" class="screen" role="main" ...>       <!-- DUPLICATE! -->
```

**Vấn đề:** Mỗi lần chỉ có 1 screen active, nhưng HTML có 3 `role="main"` cùng lúc → **violate ARIA spec**. Screen reader sẽ báo "3 main regions" gây confusion.

**Giải pháp:** Chỉ set `role="main"` cho screen đang active qua JavaScript, hoặc dùng `role="region"` cho non-active screens.

### 14. 📝 Không Có Focus-Visible Styles Cho Form Inputs

**File:** `index.html:381-386`

```css
.form-group input:focus,
.form-group textarea:focus,
.form-group select:focus {
    border-color: rgba(226, 176, 74, .5);
    box-shadow: 0 0 0 3px rgba(226, 176, 74, .1);
}
```

**Vấn đề:** CSS `:focus` style hiện tại dùng chung cho cả mouse click và keyboard navigation. Theo WCAG 2.4.7, cần `:focus-visible` để chỉ hiện outline khi dùng keyboard.

**Giải pháp:**
```css
/* Giữ :focus cho mouse, thêm :focus-visible cho keyboard */
.form-group input:focus:not(:focus-visible) {
    border-color: rgba(226, 176, 74, .3);
    box-shadow: none;
}

.form-group input:focus-visible {
    border-color: rgba(226, 176, 74, .5);
    box-shadow: 0 0 0 3px rgba(226, 176, 74, .1);
    outline: 2px solid #e2b04a;
    outline-offset: 2px;
}
```

### 15. 📝 Radio Cards — Không Có Keyboard Arrow Navigation

**File:** `index.html:826-835`

document.querySelector('.radio-group')?.addEventListener('keydown', (e) => {

    if (e.key === 'ArrowRight' || e.key === 'ArrowDown') {
        e.preventDefault();
        selectRadio(cards[(current + 1) % cards.length]);

    } else if (e.key === 'ArrowLeft' || e.key === 'ArrowUp') {
        e.preventDefault();
        selectRadio(cards[(current - 1 + cards.length) % cards.length]);
        cards[(current - 1 + cards.length) % cards.length].focus();

### 17. 📝 Inline `onclick` Handlers — Không Theo Best Practice

**File:** `index.html:760, 811, 827, 831`

```html

<label onclick="selectRadio(this)">
<form onsubmit="submitRSVP(event)">
```

**Vấn đề:**
- Inline handlers **không thể remove** → gây memory leak khi screen transitions
- Không thể test các functions một cách isolated
- Không có error handling nếu function không tồn tại
- Trộn lẫn HTML với JS logic → khó maintain

**Giải pháp:** Chuyển sang `addEventListener` trong JS:
```javascript
document.querySelector('.envelope-btn').addEventListener('click', openEnvelope);
document.getElementById('rsvp-form').addEventListener('submit', submitRSVP);
```

### 18. 📝 No Content Security Policy — XSS Attack Surface

**File:** `index.html` (thiếu `<meta>` CSP)

**Vấn đề:**
- Không có CSP header/meta tag
- Nếu có XSS vector (ví dụ qua `?guest=`), attacker có thể inject arbitrary scripts
- Google Fonts và Formspree được load từ external domains — cần `connect-src` và `font-src` directives

**Giải pháp:**
```html
<meta http-equiv="Content-Security-Policy" 
      content="default-src 'self'; 
               style-src 'self' https://fonts.googleapis.com; 
               font-src https://fonts.gstatic.com; 
               connect-src https://formspree.io; 
               script-src 'self'; 
               object-src 'none'; 
               base-uri 'self';">
```

### Pattern 1: "Code Trước, Docs Sau" → Docs Nói Một Đằng, Code Làm Một Nẻo

- Không có linting, formatting, type checking
- Không có pre-commit hooks
- Không có CI/CD → code push thẳng lên production không review
- Memory leaks, XSS risks, accessibility violations đều không bị catch

### Pattern 3: "Single File = Single Responsibility" — Tất Cả Trong Một Nồi
- HTML, CSS, JS, config — tất cả trong 1 file
- Không thể test components independently
- Không thể reuse logic
- Mỗi bug fix là phải đọc 1171 dòng để tìm

### Pattern 4: "External Dependency = No Responsibility"
- Formspree xử lý backend → **không có error handling phía client**
- Google Fonts → **không có fallback font strategy**
- No offline fallback → nếu mất mạng, page trắng
